# 三省职责深度细化设计 v3

## 1. 三省总定位

v3 中，三省不再承担“形式化流程”的职责，而是承担 **控制平面中的治理职责**。

```text
中书省：把需求变成可确认、可审核、可执行的方案。
门下省：把风险挡在交付之前，给出明确裁断。
尚书省：把任务从入口调度到闭环，并维护状态和回流。
```

任务路由器仍是唯一入口，可作为尚书省的前置模式，也可在系统成熟后拆成独立 Agent。

## 2. 三省运行关系

```text
任务路由器
  ↓
判断任务等级 / 风险 / 确认事项 / 门禁
  ↓
判断是否需要中书省方案
  ↓
判断是否需要门下省审核
  ↓
尚书省调度执行
  ↓
工部 / 刑部 / 六部模式 / Skills 执行
  ↓
失败回流 / 终审 / 交付
```

三省不是固定串行，而是按等级和风险动态组合。

## 3. 任务路由器

### 定位

任务路由器是所有任务的入口节点，负责把自然语言需求转成结构化执行判断。

### 必须判断

```text
1. 原始需求是什么；
2. 任务等级 L1 / L2 / L3 / L4；
3. 定级依据；
4. 风险类型；
5. 是否需要 A / B / C / none 确认；
6. 是否需要中书省；
7. 是否需要门下省；
8. 是否需要六部模式；
9. 是否需要技术 Skill；
10. 允许修改范围和禁止修改范围；
11. 需要哪些工程门禁；
12. 下一步状态。
```

### 禁止事项

```text
不得修改用户原始需求；
不得直接写代码；
不得跳过 A 级确认；
不得为了省事降低等级；
不得为了形式提高等级；
不得把低风险任务强行拉入完整治理。
```

### 输出

```md
# 任务路由结果 v3

## 原始需求

## 任务等级
L1 / L2 / L3 / L4

## 定级依据

## 风险类型

## 确认等级
A / B / C / none

## 需要启用的 Agent / Mode / Skill

## 不启用项及原因

## 允许修改范围

## 禁止修改范围

## 工程门禁

## 下一步状态
execute / clarify / design / review / block
```

## 4. 中书省

### 定位

中书省负责：需求澄清、范围定义、方案设计、接口 / 数据契约、技术选型、任务拆解、验收标准。

中书省不负责：写代码、上线、测试结论、最终审核、任务状态调度。

### 介入条件

| 情况 | 是否介入 |
|---|---|
| L1 且需求清楚 | 不介入 |
| L1 但需求不清 | 轻量澄清 |
| L2 普通功能 | 输出简案或契约 |
| L2 涉及 API / 数据 / 权限 | 必须介入 |
| L3 | 必须完整方案 |
| L4 | 必须里程碑方案 |
| 技术选型变化 | 必须介入 |
| 架构边界变化 | 必须介入 |
| 数据模型变化 | 必须介入 |

### 输出分层

#### 轻量澄清

```md
# 中书省轻量澄清 v3

## 原始需求
## 当前理解
## 本次范围
## 非本次范围
## 待确认问题
## 是否可以执行
```

#### L2 简案

```md
# 中书省简案 v3

## 原始需求
## 目标
## 范围
## 非范围
## 业务流程
## API / 数据契约
## 权限影响
## 验收标准
## 风险点
## 是否提交门下省审核
```

#### L3 / L4 完整方案

```md
# 中书省完整方案 v3

## 原始需求与确认记录
## 目标与非目标
## 用户 / 业务场景
## 技术选型
## 架构影响
## 模块边界
## API 契约
## 数据模型与迁移
## 权限与安全
## 部署与回滚影响
## 测试计划
## 阶段拆分
## A 级确认事项
## 提交门下省审核的问题
```

### 硬性规则

```text
不得把 L1 复杂化；
不得跳过 L3 完整方案；
不得用现有技术专家反向限制技术选型；
不得把未确认需求写成已确认；
不得替门下省做审核结论；
不得替工部写实现代码。
```

## 5. 门下省

### 定位

门下省负责审核、驳回和风险裁断。

```text
低风险快速放行；
中风险标准审核；
高风险严格阻断；
结论必须明确。
```

### 审核触发条件

#### L1

默认不触发门下省，除非：发现误判、涉及安全 / 权限 / 数据 / 部署隐患、刑部认为验证不足、用户要求严格审核。

#### L2

以下情况必须审核：

```text
API 契约变化；
权限判断；
多个模块联动；
用户可见核心功能；
刑部输出有条件可交付；
工部声明残余风险；
文档与实现可能不一致；
用户要求终审。
```

#### L3 / L4

必须完整审核和阶段审核。

### 输出

```md
# 门下省审核报告 v3

## 审核对象
## 任务等级
## 风险等级
低 / 中 / 高 / 极高
## 审核模式
轻量 / 标准 / 完整
## 审核结论
通过 / 有条件通过 / 驳回
## 阻塞问题
## 非阻塞问题
## 必须修改项
## 建议修改项
## 可延期事项
## 是否需要皇帝确认
## 是否允许进入下一步
```

### 驳回规则

驳回必须同时满足：指出具体问题、标明风险等级、说明为什么阻塞、给出必须修改项、指定退回对象、说明是否允许重新提交。

门下省不得因为模板不够完整驳回本可轻量处理的 L1 / 低风险 L2。

## 6. 尚书省

### 定位

尚书省负责调度、状态管理、并行协作、失败回流和交付汇总。

### 核心职责

```text
1. 接收任务路由结果；
2. 建立任务状态；
3. 分派 Agent / Mode / Skill；
4. 控制串行和并行任务；
5. 维护允许修改范围；
6. 维护禁止修改范围；
7. 收集门禁结果；
8. 处理失败回流；
9. 汇总执行、验证、文档和风险；
10. 提交终审或交付。
```

## 工程门禁

## 7. 三省一句话定义

```text
任务路由器定入口，中书省定方案，门下省定风险，尚书省定执行；状态机定流转，工程门禁定能否交付。
```

## 19. 三省对数据存储边界的职责

### 中书省

中书省在涉及数据的方案中必须明确：

```text
数据类型；
数据分级；
主存储位置；
允许副本；
禁止位置；
是否向量化；
是否脱敏；
保留期限；
删除策略；
迁移和回滚方案。
```

### 门下省

门下省必须审核：

```text
是否把 Secret、生产数据、隐私、长日志、scratchpad 写入了错误位置；
是否过度采集；
是否错误向量化；
是否缺少删除和保留策略；
是否绕过 Secret Manager / GitHub Secrets；
是否存在不可审计的数据副本。
```

### 尚书省

尚书省调度时必须明确：

```text
允许写入的位置；
禁止写入的位置；
需要保留的证据；
只存摘要还是存原文；
artifact 存储路径；
史官记录级别；
是否需要户部 / 兵部 / 门下省介入。
```
