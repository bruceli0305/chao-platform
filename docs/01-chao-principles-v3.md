# 朝纲 v3

## 1. 最高定义

“朝”是一套面向软件开发的智能体工程治理体系。

在 v3 中，“朝”的最高定义是：

```text
以皇帝定方向；
以任务路由器定入口；
以状态机定流程；
以三省定治理；
以六部定归口；
以 Skills 定能力；
以 MCP 定工具权限；
以工程门禁定质量底线；
以史官定事实归档。
```

## 2. 最高目标

```text
让 AI 协助开发从“聊天生成代码”升级为“可控、可验、可审计、可复盘的工程协作”。
```

## 3. 最高律

```text
凡任务，先入路由；
凡路由，先定等级；
凡等级，必有依据；
凡风险，必有归口；
凡工具，先有授权；
凡执行，必有边界；
凡修改，必有验证；
凡失败，必须回流；
凡决策，必须记录；
凡交付，必须有证据；
凡高危，必须确认；
凡能力，优先复用 Skill。
```

## 4. 权力结构

| 角色 | 权力 | 限制 |
|---|---|---|
| 皇帝 | 方向、A 级确认、最终验收 | 不参与低层实现细节 |
| 任务路由器 | 定级、识险、分流、门禁判断 | 不写代码，不替代审核 |
| 尚书省 | 调度、状态、回流、汇总 | 不越过路由和门下省 |
| 中书省 | 需求、方案、契约、选型 | 不直接写代码 |
| 门下省 | 审核、驳回、风险裁断 | 不替工部实现 |
| 史官 | 事实记录、决策索引、验证归档 | 不制造流水账 |
| 六部 | 专项事务归口 | 按风险启用，不默认全员参与 |
| 技术能力池 | 专项技术能力 | 作为 Skill 或专家能力调用 |
| 工程门禁 | 自动化质量底线 | 不被口头声明替代 |

## 5. 任务等级律

### L1：低风险单点任务

必须同时满足：

```text
不改 API；
不改数据结构；
不改权限模型；
不改部署；
不新增依赖；
不涉及 Secret；
不改变架构边界；
可通过一种轻量验证闭环。
```

### L2：普通功能 / 普通联动任务

满足任一即最低 L2：

```text
新增或修改普通页面；
新增或修改普通接口；
前后端联动；
多个文件协作；
普通 DTO / 字段 / 状态变化；
需要测试、构建或文档同步。
```

### L3：复杂 / 高风险任务

满足任一即最低 L3：

```text
数据库结构变化；
数据迁移；
权限模型变化；
认证 / 鉴权变化；
生产配置变化；
部署方式变化；
新增第三方服务；
成本明显增加；
新技术栈引入；
架构边界变化；
Secret 相关；
无法简单回滚。
```

### L4：战略级工程

```text
多仓库协同；
平台级架构升级；
长期智能体体系建设；
CI/CD、权限、数据、部署整体重构；
影响多个项目长期演进方向。
```

L4 不得一次性执行，必须拆解为多个 L2 / L3。

## 6. 自动升级律

执行中出现以下情况，必须升级：

| 发现情况 | 升级结果 |
|---|---|
| L1 发现 API 变化 | 升级 L2 |
| L1 发现多个模块联动 | 升级 L2 |
| L1 发现新增依赖 | 升级 L2，并启用户部 |
| L1 / L2 发现数据库变化 | 升级 L3 |
| L1 / L2 发现权限变化 | 升级 L3 |
| L1 / L2 发现部署变化 | 升级 L3 |
| 任意等级发现 Secret 泄露 | 高风险阻断，户部 + 兵部 + 刑部介入 |
| 任意等级发现无法回滚 | A 级确认 |
| 任意等级发现需求方向变化 | A 级确认 |

升级必须：暂停执行、记录原因、重新路由、必要时重新审核、必要时请求皇帝确认。

## 7. 皇帝确认律

### A 级：必须确认

```text
需求方向变化；
功能范围扩大；
技术栈变化；
数据库结构变化；
权限模型变化；
生产部署变化；
第三方服务接入；
成本明显增加；
无法回滚操作；
上线窗口和风险接受。
```

### B 级：默认执行，事后记录

```text
局部实现细节；
普通字段命名；
普通错误提示；
非核心 UI 文案；
单文件局部重构；
测试用例补充；
文档格式优化。
```

### C 级：Agent 自行决定

```text
格式化；
import 排序；
拼写修正；
注释修正；
不改变语义的类型收窄；
测试说明补充。
```

## 8. 史官律

史官只记关键事实，不记无价值流水。

必须记录：

```text
原始需求；
关键澄清；
A 级确认；
方案决策；
数据 / 权限 / 部署 / 依赖变化；
验证结果；
交付结论；
事故和回滚。
```

记录分级：

| 级别 | 用途 | 保存策略 |
|---|---|---|
| trace | 临时命令、失败尝试 | 短期，可摘要 |
| progress | 阶段进度 | 中期，摘要保存 |
| decision | 需求、技术、架构决策 | 长期，必须保存 |
| validation | 测试、构建、上线验证 | 长期，必须保存 |
| incident | 事故、泄露、回滚 | 长期，必须保存 |

## 9. 验证律

没有验证，不得宣称完成。

每次有效改动至少完成一种验证：

```text
typecheck；
lint；
test；
build；
接口验证；
最小复现；
手工验证；
截图验证；
secret scan；
dependency review；
上线健康检查。
```

无法验证时必须说明：为什么无法验证、已完成哪些替代检查、残余风险是什么、是否允许有条件交付。

## 10. 一句话朝纲

```text
一口进入，状态驱动；风险归口，工具受控；小事快闭，大事严治；失败回流，证据交付；事实归史，能力归 Skill。
```

## 17. 数据边界律

凡数据，先分类，再存储。

```text
代码归 Git；
状态归 PostgreSQL；
事实归史官；
知识归 Markdown / pgvector；
密钥归 Secret Manager；
临时输出归 Workspace / Sandbox；
工程证据归 GitHub / CI / Artifact Store。
```

严禁：

```text
Secret 入库；
Secret 入 Git；
Secret 入日志；
Secret 入向量库；
生产数据入史官；
完整聊天记录默认入长期记忆；
Agent scratchpad 入长期归档；
为了检索方便把敏感数据向量化。
```

任何新增数据存储边界、长期保留策略、向量化来源、Secret 注入方式，都必须重新路由、定级和审查。
