# 工程门禁与质量体系 v3

## 1. 最高原则

```text
没有验证，不得完成；
门禁失败，不得交付；
无法验证，必须说明；
风险接受，必须确认；
质量结论，必须明确。
```

## 2. 基础门禁

| 门禁 | 适用 |
|---|---|
| typecheck | TypeScript / 前端 / Node 项目 |
| lint | 代码风格和静态检查 |
| test | 单元 / 集成 / 回归测试 |
| build | 构建验证 |
| secret_scan | Secret 泄露检查 |
| dependency_review | 依赖变更审查 |
| migration_test | 数据库迁移验证 |
| healthcheck | 部署后健康检查 |
| manual_validation | 手工验证或截图验证 |

## 3. 任务类型门禁矩阵

| 任务类型 | 必须门禁 |
|---|---|
| L1 文案 / 样式 | 手工验证或截图验证 |
| L1 类型修复 | typecheck |
| L2 前端功能 | typecheck + lint + build + 手工验证 |
| L2 API | test + interface validation |
| L2 前后端联动 | build + 接口联调 + 回归验证 |
| 新增依赖 | dependency review + audit |
| 数据库变更 | migration test + rollback plan |
| 权限变化 | 权限测试 + 越权测试 |
| 部署变化 | build artifact + healthcheck + rollback |
| Secret 相关 | secret scan + 密钥注入检查 |
| L3 / L4 | 测试计划 + 阶段验证 + 门下省审核 |

## 4. 禁止合入条件

```text
测试失败；
构建失败；
Secret 扫描失败；
未说明新增依赖原因；
L3 任务没有回滚方案；
A 级事项没有确认；
工部声明未验证却请求完成；
刑部结论为不可交付；
门下省驳回仍继续推进。
```

## 5. 刑部质量结论

刑部只能输出：

```text
可交付；
有条件可交付；
不可交付。
```

有条件可交付必须说明：

```text
未覆盖内容；
残余风险；
建议上线条件；
是否需要皇帝确认；
是否需要门下省终审。
```

## 6. PR Checklist

```md
# PR Checklist v3

- [ ] 已关联任务编号。
- [ ] 已完成任务定级。
- [ ] 已识别风险类型。
- [ ] 未涉及 A 级事项，或已完成确认。
- [ ] 修改范围符合任务要求。
- [ ] 未修改无关文件。
- [ ] 未新增未审查依赖。
- [ ] 已完成必要验证。
- [ ] 已说明未覆盖内容。
- [ ] 已说明残余风险。
- [ ] 需要记录的事项已同步史官。
- [ ] 刑部质量结论明确。
- [ ] 门下省终审已按触发条件执行。
```

## 7. 验证证据格式

```md
# 验证结果

## 验证命令

## 执行环境

## 结果
通过 / 失败 / 未执行

## 输出摘要

## 未覆盖内容

## 残余风险

## 质量结论
可交付 / 有条件可交付 / 不可交付
```

## 8. 数据边界门禁

涉及数据存储、日志、Secret、向量化、artifact 的任务，必须增加数据边界门禁：

```text
Secret Scan；
.env 提交检查；
日志脱敏检查；
artifact 内容检查；
pgvector ingest 白名单检查；
数据库 schema review；
数据分级检查；
存储位置检查；
保留期限检查。
```

质量结论中必须明确：

```text
是否存在越界存储；
是否存在敏感信息入库；
是否存在敏感信息入 Git；
是否存在敏感信息入向量库；
是否存在未授权 artifact；
是否存在未定义保留期限的数据。
```
