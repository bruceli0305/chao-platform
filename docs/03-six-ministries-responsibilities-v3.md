# 六部职责深度细化设计 v3

## 1. 六部总定位

六部是专项事务归口体系，不是默认全量参与体系。

```text
吏部防膨胀；
户部管资源；
礼部管表达；
兵部管上线；
刑部管验证；
工部管实现。
```

v3 进一步明确：

```text
六部按风险启用；
默认只启用工部 / 刑部；
其他部门优先作为 Mode；
高风险任务才多部门协同；
不启用也要说明理由；
输出按等级裁剪，不制造文档噪音。
```

## 2. 六部常驻与按需

| 部门 | v3 形态 | 是否常驻 | 说明 |
|---|---|---|---|
| 工部 | Agent | 常驻 | 任何代码实现或修复必启用 |
| 刑部 | Agent | 常驻 | Bug、新功能、权限、安全、回归验证必启用 |
| 吏部 | Mode | 按需 | Agent、Prompt、规范、越权治理 |
| 户部 | Mode | 按需 | 依赖、配置、数据库、成本、Secret |
| 礼部 | Mode | 按需 | README、API 文档、发布说明、UI 文案 |
| 兵部 | Mode | 按需 | 部署、CI/CD、Docker、Nginx、回滚、应急 |

## 3. 启用硬规则

| 情况 | 必须启用 |
|---|---|
| 任何代码实现 | 工部 |
| Bug 修复 | 刑部 + 工部 |
| 新功能验证 | 刑部 |
| 新增 / 删除 / 升级依赖 | 户部 |
| lockfile 变化 | 户部 |
| 新增 / 修改环境变量 | 户部 + 兵部 |
| 数据库结构变化 | 中书省 + 户部 + 门下省 |
| 数据迁移 | 中书省 + 户部 + 兵部 + 刑部 + 门下省 |
| Secret 相关 | 户部 + 兵部 + 刑部 |
| 正式 README / API 文档 / 发布说明 | 礼部 |
| 部署 / CI/CD / Nginx / Docker | 兵部 |
| 生产配置变化 | 兵部 + 门下省 |
| Agent / Prompt / 规范变化 | 吏部 |
| 权限 / 安全审计 | 刑部 + 门下省 |

## 4. 不启用理由制度

尚书省调度时必须列出不启用的部门及原因。

示例：

```md
## 不启用项及原因

- 吏部：不涉及 Agent、Prompt、规范变化。
- 户部：不涉及依赖、配置、数据库、Secret 或成本变化。
- 礼部：不涉及正式文档、用户说明或发布说明。
- 兵部：不涉及部署、CI/CD、生产配置或回滚。
```

## 5. 六部通用输入标准

六部接任务前必须看到：

```text
任务编号；
原始需求；
任务等级；
风险类型；
本部门目标；
输入材料；
允许修改范围；
禁止修改范围；
验收标准；
工程门禁；
回交对象。
```

缺失处理：

```text
L1：可自行补足最小必要信息并说明假设；
L2：请求尚书省补充；
L3 / L4：不得执行，必须补齐。
```

## 6. 吏部 v3

### 定位

吏部负责 Agent、Mode、Skill、Prompt、AGENTS.md、工程规范、职责边界和越权治理。

### 启用条件

```text
新增长期 Agent；
修改 Agent 职责；
修改 Prompt 模板；
修改 AGENTS.md；
修改工程规范；
新增 Skill 或修改 Skill 边界；
出现职责冲突；
出现越权行为。
```

### 输出

```md
# 吏部治理结果 v3

## 涉及角色 / Mode / Skill / 规范
## 是否需要新增 Agent
## 是否可改为 Mode 或 Skill
## 新增或修改内容
## 职责边界
## 越权风险
## 是否更新 AGENTS.md
## 是否需要史官记录制度变更
```

## 7. 户部 v3

### 定位

户部负责依赖、配置、数据库、成本、Secret、许可证、外部服务资源。

### 启用条件

```text
新增 / 删除 / 升级依赖；
修改 lockfile；
新增 / 修改环境变量；
修改数据库结构；
涉及数据迁移；
涉及真实密钥或 Token；
接入第三方 API；
引入云资源或付费服务；
许可证风险。
```

### 门禁

| 风险 | 工具 / 证据 |
|---|---|
| npm 依赖风险 | npm audit |
| Rust 依赖风险 | cargo audit |
| Secret 泄露 | gitleaks / GitHub secret scanning |
| 依赖变更 | Dependency Review |
| 许可证风险 | license checker |
| 成本变化 | 成本说明 / 配额说明 |

### 输出

```md
# 户部审查结果 v3

## 审查范围
## 依赖变化
## 配置变化
## 数据库 / 数据资源影响
## Secret 风险
## 成本影响
## 许可证风险
## 必须修改项
## 是否允许继续
```

## 8. 礼部 v3

### 定位

礼部负责正式表达、接口文档、用户说明、发布说明、UI 文案一致性。礼部不得美化失败。

### 启用条件

```text
README；
API 文档；
用户手册；
发布说明；
正式功能说明；
UI 文案统一；
文档与实现一致性校对。
```

### 输出

```md
# 礼部文档结果 v3

## 文档类型
## 面向对象
## 依据材料
## 功能说明
## 操作 / 接口说明
## 已知问题
## 未完成事项
## 一致性校验结论
```

## 9. 兵部 v3

### 定位

兵部负责部署、CI/CD、Docker、Nginx、生产配置、安全暴露面、回滚、应急。

### 启用条件

```text
部署方式变化；
生产配置变化；
Nginx / Docker / CI/CD 变化；
环境变量上线注入；
数据库迁移上线；
需要回滚方案；
安全暴露面变化；
线上故障。
```

### 输出

```md
# 兵部执行结果 v3

## 部署目标
## 配置项
## CI/CD
## 安全检查
## 健康检查
## 回滚方案
## 上线后观察项
## 风险点
## 是否建议上线
```

## 10. 刑部 v3

### 定位

刑部负责测试、审计、Bug 根因、异常路径、权限边界、回归验证和质量结论。

### 启用条件

```text
Bug 修复；
新功能验证；
权限安全审计；
回归测试；
工部完成实现后需要质量结论；
门下省要求补充验证。
```

### 质量结论

刑部只能输出：

```text
可交付；
有条件可交付；
不可交付。
```

不得输出：

```text
应该可以；
大概没问题；
问题不大；
先合了再说。
```

### 输出

```md
# 刑部验证结果 v3

## 测试范围
## 验证方式
## Bug 现象 / 根因
## 正常路径
## 异常路径
## 权限 / 边界
## 回归验证
## 未覆盖内容
## 残余风险
## 质量结论
```

## 11. 工部 v3

### 定位

工部负责代码实现、Bug 修复、局部重构和最小可验证改动。

### 核心要求

```text
最小；
准确；
可验证；
不越界；
不假成功。
```

### 硬性约束

```text
不擅自修改需求；
不擅自扩大改动范围；
不擅自改变技术选型；
不擅自修改生产配置；
不顺手重构；
不使用 any、空对象、默认值、静默 catch 掩盖问题；
不吞异常；
不返回假成功；
不引入未审查依赖；
不无验证宣称完成。
```

### 输出

```md
# 工部实现结果 v3

## 任务目标
## 修改文件
## 新增文件
## 核心逻辑
## 未修改范围
## 错误处理 / 日志
## 验证结果
## 未完成事项
## 风险点
```

## 12. 六部一句话定义

```text
工部做实现，刑部证质量，户部控资源，兵部守上线，礼部保一致，吏部防膨胀。
```

## 30. 六部对数据存储边界的职责

```text
吏部：维护数据边界规则、Agent 权限和越权处理。
户部：审查数据库、配置、Secret、存储资源、成本和数据分级。
礼部：确保文档和发布说明不泄露敏感数据。
兵部：管理环境变量注入、CI 日志、artifact、部署配置和回滚数据。
刑部：验证 Secret Scan、日志脱敏、权限访问和数据边界门禁。
工部：实现时不得把数据写入错误存储位置，不得为调试方便泄露敏感信息。
```

凡涉及新增 Secret、数据库表、向量索引来源、对象存储、日志保留策略、生产数据处理，必须启用户部；涉及部署环境、CI、artifact、日志和回滚时，必须启用兵部。
